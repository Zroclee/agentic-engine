这是一个非常核心且关键的架构设计问题。在 Agentic as a Service (AaaS) 平台中，平台需要代理 AI 智能体去调用客户的业务 API。为了保证数据安全、防止凭证泄漏，我们需要在**数据库存储**、**网关代理层**以及**前端展示**三个维度进行严密的设计。

由于平台必须要在某个环节携带凭证去请求用户的 API，所谓的“不保存用户的接口信息”，在实际工程落地时，通常转化为：**平台不保存明文凭证，不在前端暴露凭证，且只在后端内存中瞬时解密使用。**

以下是为您推荐的项目管理模块 Auth 认证字段设计方案及安全架构：

### 一、 数据库字段设计 (Prisma Schema)

建议在 `Project`（项目）或 `Tool`（工具）模型上增加灵活的认证配置字段。考虑到同一个项目下的不同 API 可能共享同一个鉴权方式，建议在 `Project` 级别配置基础 Auth，同时允许 `Tool` 级别覆盖。

在您的 `schema.prisma` 中，可以这样设计：

```prisma
// 认证类型枚举
enum AuthType {
  NONE            // 无需认证：适用于公开或基于IP白名单的接口
  API_KEY         // API Key认证：静态密钥，随请求携带
  DYNAMIC_TOKEN   // 动态Token认证：使用secret调用鉴权接口换取临时token
}

model Project {
  id          String   @id @default(uuid())
  name        String
  description String?
  
  // --- 新增的认证相关字段 ---
  authType    AuthType @default(NONE)
  
  // 核心：将认证所需的所有敏感信息序列化为 JSON 后进行 AES-256-GCM 强加密。
  // - API_KEY 存储: { "key": "sk-xxx", "position": "header", "keyName": "Authorization" }
  // - DYNAMIC_TOKEN 存储: { "secret": "xxx", "tokenUrl": "https://...", "tokenPath": "data.token" }
  authConfig  String?  @db.Text 
  
  tools       Tool[]
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

model Tool {
  id          String   @id @default(uuid())
  projectId   String
  project     Project  @relation(fields: [projectId], references: [id])
  
  name        String
  apiUrl      String   // 用户提供的接口地址 (建议在后端强制校验必须为 https)
  
  // 允许工具级别覆盖项目的默认认证
  overrideAuth Boolean @default(false)
  authType     AuthType?
  authConfig   String?  @db.Text

  // ...
}
```

### 二、 核心安全机制设计

为了确保“平台安全访问，不导致数据泄漏”，必须遵循以下四个核心原则：

#### 1. 凭证落库：AES 对称加密 (Encryption at Rest)
用户的 API Key 或 Token 绝对不能以明文存入 PostgreSQL。在 NestJS 后端接收到用户配置的凭证时，必须使用服务器的全局密钥（例如配置在 `.env` 中的 `ENCRYPTION_KEY`）对其进行加密。

**NestJS 加密服务示例：**
```typescript
import { Injectable } from '@nestjs/common';
import * as crypto from 'crypto';

@Injectable()
export class CryptoService {
  private readonly algorithm = 'aes-256-gcm';
  // 从环境变量获取 32 字节的密钥
  private readonly secretKey = process.env.ENCRYPTION_KEY; 

  encrypt(text: string): string {
    const iv = crypto.randomBytes(16);
    const cipher = crypto.createCipheriv(this.algorithm, Buffer.from(this.secretKey, 'hex'), iv);
    let encrypted = cipher.update(text, 'utf8', 'hex');
    encrypted += cipher.final('hex');
    const authTag = cipher.getAuthTag().toString('hex');
    // 拼接格式：iv:authTag:encryptedData
    return `${iv.toString('hex')}:${authTag}:${encrypted}`;
  }

  decrypt(hash: string): string {
    const parts = hash.split(':');
    const iv = Buffer.from(parts[0], 'hex');
    const authTag = Buffer.from(parts[1], 'hex');
    const encryptedText = Buffer.from(parts[2], 'hex');
    const decipher = crypto.createDecipheriv(this.algorithm, Buffer.from(this.secretKey, 'hex'), iv);
    decipher.setAuthTag(authTag);
    let decrypted = decipher.update(encryptedText, 'hex', 'utf8');
    decrypted += decipher.final('utf8');
    return decrypted;
  }
}
```

#### 2. 前端展示：脱敏与单向写入 (Data Masking)
当用户在项目管理页面查看已配置的 Auth 信息时，**后端永远不要返回明文或密文的凭证**。
*   **读取时**：如果 `authConfig` 有值，后端返回给前端类似 `{"token": "sk-****1234"}` 的脱敏数据，或者只返回一个布尔值 `hasAuth: true`。
*   **修改时**：用户如果需要修改，只能重新输入全新的 Token 进行覆盖，不能基于原有的 Token 进行编辑。

#### 3. 运行机制与动态 Token 缓存 (Backend Proxy & Token Lifecycle)
**绝对不要在前端直接调用用户的 API**，这会导致跨域问题且极不安全。
针对这三种认证方式，当 AI 决定调用某个工具时，网关代理的执行流程如下：

1. **路由拦截与信息获取**：NestJS 收到调用指令，根据 `Tool ID` 获取所属 `Project` 的 `authType` 和密文 `authConfig`。
2. **凭证处理与缓存机制 (Redis)**：
   - **NONE**: 直接跳过认证。
   - **API_KEY**: 内存中瞬间解密出静态 Key，直接注入请求。
   - **DYNAMIC_TOKEN (核心逻辑)**: 
     - **查缓存**：先在 Redis 查询 `project:{id}:auth_token`。
     - **未命中/已过期**：解密 `authConfig` 得到 `secret` 和 `tokenUrl` -> 发起 HTTP 请求换取 Token -> 依据配置的 `tokenPath` 提取 Token -> 存入 Redis 并设置合理的 TTL。
     - **并发控制**：多个并发请求同时发现 Token 过期时，需使用 Redis 分布式锁，确保只有一个请求去执行换取动作，其余等待。
3. **发起业务请求**：由 NestJS 服务器作为客户端，携带最终的 Token 或 Key 向用户的 API 发起请求。
4. **401 容错与重试**：如果业务接口返回 `401 Unauthorized`，网关需立即**失效对应的 Redis Token 缓存**，并自动重试一次“换取 Token -> 重新调用接口”的完整流程。
5. **安全回收**：获取业务数据后，明文凭证（secret/key）在内存中立即被垃圾回收（GC）。

#### 4. 网络层防护 (Network Security)
*   **强制 HTTPS**：在保存工具 API 地址时，后端校验必须是以 `https://` 开头，防止凭证在公网传输时被抓包。
*   **提供平台固定出口 IP**：在企业级 AaaS 场景中，建议为您的网关服务配置固定的公网出口 IP（如使用 NAT 网关）。这样您可以告诉用户：“请在您的防火墙中将 IP `x.x.x.x` 加入白名单”。这是防止用户 API 被恶意盗刷的最后一道防线。

### 三、 交互流程总结

1. **用户配置时**：前端表单选择 `Bearer Token` -> 填入 `sk-xxxx` -> 提交给 NestJS -> NestJS 加密存入 DB。
2. **用户查看时**：前端请求项目详情 -> NestJS 发现数据库有值 -> 返回 `authType: BEARER_TOKEN, isConfigured: true` (不返回真实 token) -> 前端显示 "已配置 (重新配置)"。
3. **AI 对话时**：AI 触发工具 -> NestJS 拦截请求 -> 从 DB 取出密文解密 -> 组装 `Authorization: Bearer sk-xxxx` 发送请求到客户系统 -> 客户系统校验通过并返回业务数据 -> NestJS 丢弃明文 Token，将业务数据送入 AI 上下文。
