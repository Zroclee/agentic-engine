为了保证超级管理员和其他用户配置的 **API Key 安全**，这是一个非常关键且经典的系统安全设计问题。

我们绝不能将 API Key 明文存储在数据库中，也不能在前端随意暴露。以下是标准的业界安全实践方案：

### API Key 核心安全保障方案

1. **应用层对称加密 (AES-256-GCM)**：
   - **入库前加密**：无论是超级管理员还是普通用户填写的 API Key，在网关（Gateway）保存到数据库之前，必须使用 `AES-256-GCM` 算法进行强加密。
   - **密钥隔离**：加密用的主密钥（Master Key）存放在服务器的 `.env` 环境变量中，绝对不提交到代码仓库。这样即使数据库被黑客拖库，他们拿到的大模型表里的 `apiKey` 也只是一堆无法还原的乱码。
2. **脱敏与单向传输 (Masking)**：
   - **绝不明文返回**：当用户（甚至是创建者本人）在前端查看大模型列表或详情时，后端返回的 API Key 必须是脱敏的（例如：`sk-****1234`）。
   - **修改逻辑**：如果用户要修改 API Key，前端提交新的 Key，后端直接覆盖加密存储；如果不修改，前端提交空值或占位符，后端不更新此字段。
3. **网关代理请求 (Proxy)**：
   - 前端（如 chat-web）永远不要直接使用 API Key 去请求 OpenAI 或其他大模型厂商。
   - 所有的对话请求都发送给 Gateway 网关。网关根据用户选择的模型 ID，在内存中查出密文，解密出明文 API Key，并在服务端向大模型厂商发起请求。**明文 Key 的生命周期仅存在于网关服务单次请求的内存中**。
4. **数据权限隔离 (RBAC)**：
   - 在查询可用大模型列表时，后端强制加上过滤条件：`is_public = true`（公共模型） 或 `user_id = 当前登录用户ID`（私有模型）。
   - 只有具备“超级管理员”角色的用户，在新增大模型时，才有权限将 `isPublic` 字段设为 `true`。
